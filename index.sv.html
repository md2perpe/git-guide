skapa en ny repository

skapa en ny katalog, öppna den och kör
git init
för att skapa ett nytt git-repository.

hämta ut en repository

skapa en arbetskopia av en lokal repository genom att köra kommandot
git clone /sökväg/till/repository
när du använder en fjärrserver, blir ditt kommando
git clone username@host:/sökväg/till/repository

arbetsflöde

din lokala repository består av tre "träd" hanterade av git. det första är din Arbetskatalog som innehåller själva filerna. det andra är Index som fungerar som ett avlastningsområde och slutligen HEAD som pekar på din senaste commit.

add & commit

Du kan föreslå ändringar (lägga det till Index) genom
git add <filnamn>
git add *
Detta är det första steget i det grundläggande arbetsflödet för git. För att verkligen skicka in dessa ändringar används
git commit -m "Meddelande"
Nu har filen skickats till HEAD, men finns inte i din fjärrepository än.

skicka upp ändringar

Dina ändringar finns nu i HEAD i din lokala arbetskopia. För att skicka dessa ändringar till din fjärrepository, kör
git push origin master
Ändra master till den gren (branch) du vill skicka dina ändringar till.

Om du inte har klonat en existerande repository och vill ansluta din repository till en fjärrserver, behöver du lägga till den med
git remote add origin <server>
Nu kan du skicka upp dina ändringar till den valda fjärrservern

förgrening

Grenar (branch) används för att utveckla funktionaliteter isolerade från varandra. Grenen master är standardgrenen när du skapar en repository. Använd andra grenar för utveckling och sammanfoga dem tillbaka till master när du är klar.

skapa en ny gren kallad "feature_x" och byt till den genom
git checkout -b feature_x
byt tillbaka till master
git checkout master
och radera grenen igen
git branch -d feature_x
en gren är inte tillgänglig för andra förrän du har skickat upp ändringarna till din fjärrepository
git push origin <branch>

update & merge

för att uppdatera din lokala repository till den nyaste commiten, kör
git pull
in din arbetskatalog för att hämta and införliva ändringar från fjärrepositoryn.
för att införliva en annan gren in i din aktiva gren (t.ex. master), använd
git merge <branch>
i båda fallen försöker git auto-införliva ändringarna. Tyvärr är detta inte alltid möjligt utan resulterar i konflikter. Du måste då manuellt införliva dessa konflikter genom att redigera filerna som git visar. Efter ändring behöver du markera dem som införlivade med
git add <filename>
innan du införlivar ändringar kan du titta på dem genom
git diff <source_branch> <target_branch>

tagging

it's recommended to create tags for software releases. this is a known concept, which also exists in SVN. You can create a new tag named 1.0.0 by executing
git tag 1.0.0 1b2e1d63ff
the 1b2e1d63ff stands for the first 10 characters of the commit id you want to reference with your tag. You can get the commit id with
git log
you can also use fewer characters of the commit id, it just has to be unique.

replace local changes

In case you did something wrong (which for sure never happens ;) you can replace local changes using the command
git checkout -- <filename>
this replaces the changes in your working tree with the last content in HEAD. Changes already added to the index, as well as new files, will be kept.

If you instead want to drop all your local changes and commits, fetch the latest history from the server and point your local master branch at it like this
git fetch origin
git reset --hard origin/master

useful hints

built-in git GUI
gitk
use colorful git output
git config color.ui true
show log on just one line per commit
git config format.pretty oneline
use interactive adding
git add -i
